# [PRD] DotAgent Office: 픽셀 아트 에디션
## AI 에이전트 협업 시뮬레이터

---

## 1. 제품 개요 (Executive Summary)

### 1.1 비전
사용자가 부여한 복잡한 임무를 여러 명의 AI 에이전트가 협업하여 해결하는 과정을 **8비트 복고풍 픽셀 아트 환경**에서 시각화합니다. 단순 텍스트 결과가 아닌, 에이전트들이 사무실 내에서 움직이고 상호작용하는 모습을 통해 **직관적인 진행도 인식**과 **협업의 느낌**을 전달합니다.

### 1.2 핵심 가치 제안
- **직관적 시각화**: 텍스트 로그보다 **픽셀 애니메이션**으로 더 빠른 이해
- **감정적 연결**: 고유한 개성을 가진 에이전트 캐릭터로 **인간적 느낌**
- **몰입도**: 8비트 게임 같은 레트로 감성으로 **사용자 집중도 상승**
- **접근성**: 의존성 없이 순수 React만으로 **모든 환경에서 동작**

### 1.3 타겟 사용자
1. **개발자**: Multi-agent LLM 워크플로우 모니터링 필요
2. **PM/기획자**: AI 협업 과정의 시각적 이해 필요
3. **학생/연구자**: LLM 에이전트 시뮬레이션 학습 목적
4. **크리에이터**: 재미있는 UI/UX 실험 관심층

## 2. 문제 정의 (Problem Statement)

### 2.1 현재 상황의 문제점
```
❌ 기존 상황
- 에이전트 작업: 텍스트 로그만 출력
- 시각적 피드백 부족: 진행도 알기 어려움
- 감정적 engagement 낮음: 기술적이고 차가움
- 모니터링 불편: 여러 에이전트의 상태를 한눈에 파악 불가
```

### 2.2 해결해야 할 핵심 문제
1. **시각화 부재**: 에이전트의 위치와 상태를 직관적으로 표현할 방법이 없음
2. **상호작용 부재**: 에이전트 간의 소통을 어떻게 표현할 것인가?
3. **사용자 몰입도**: 단조로운 텍스트 출력에서 벗어나 흥미로운 UX 제공 필요
4. **프로덕션 준비 미흡**: 초기 프로토타입 단계에서 확장 가능한 아키텍처 필요

### 2.3 제품으로 달성할 목표
```
✅ 원하는 상태
- 에이전트의 위치를 실시간으로 시각화
- 각 에이전트의 상태(대기/이동/작업/협업)를 직관적으로 표현
- 협업 과정을 애니메이션으로 표현
- 사용자가 재미있게 AI 협업을 모니터링
```

## 3. 핵심 기능 (Key Features)

### 3.1 에이전트 페르소나 시스템

#### 3.1.1 에이전트 속성
각 에이전트는 다음 속성을 가짐:

```javascript
{
  id: string,                    // 고유 ID (leo, momo, alex)
  name: string,                  // 표시 이름
  role: string,                  // 역할 설명
  color: string,                 // 픽셀 아트 색상 (#ff4444)
  position: {
    zone: 'work' | 'meeting' | 'lounge',
    x: number,                   // 구역 내 X 좌표 (px)
    y: number                    // 구역 내 Y 좌표 (px)
  },
  status: AgentStatus,           // 현재 상태
  stressLevel: number,           // 스트레스 수치 (0-100)
  skillTags: string[]            // 기술 태그 (v2.0+)
}
```

#### 3.1.2 초기 에이전트 설정

| 에이전트 | 이름 | 역할 | 색상 | 특징 |
|---------|------|------|------|------|
| leo | LEO | 코드 마스터 | #ff4444 (빨강) | 개발/구현 전문가 |
| momo | MOMO | 기획 천재 | #ffaa00 (주황) | 전략/기획 전문가 |
| alex | ALEX | 분석가 | #00ccff (청록) | 분석/검증 전문가 |

#### 3.1.3 에이전트 상태 (AgentStatus)
```typescript
type AgentStatus = 
  | 'IDLE'           // 대기 상태 (기본)
  | 'MOVING'         // 이동 중 (구역 간 이동 0.6초)
  | 'WORKING'        // 작업 중 (태스크 처리)
  | 'COMMUNICATING'  // 소통 중 (다른 에이전트와 협업)
  | 'RESTING'        // 휴식 중 (라운지)
```

상태 전환 다이어그램:
```
IDLE ──(move)──> MOVING ──(arrive)──> IDLE
                              ↓
                         COMMUNICATING
                              ↓
                            IDLE
```

### 3.2 가상 오피스 월드 (Grid Map & Zones)

#### 3.2.1 구역 구조
```
┌─────────────────────────────────────────────────┐
│  ┌──────────┐  ┌──────────┐  ┌──────────┐      │
│  │  WORK    │  │ MEETING  │  │ LOUNGE   │      │
│  │  ZONE    │  │  ROOM    │  │          │      │
│  │          │  │          │  │          │      │
│  │ [■][■]  │  │   []     │  │    []    │      │
│  │          │  │          │  │          │      │
│  └──────────┘  └──────────┘  └──────────┘      │
│                                                 │
│  ┌──────────────────────────────────────────┐  │
│  │         [TERMINAL LOG]                   │  │
│  │ > Agent Leo moved to MEETING ROOM        │  │
│  │ > Agent Momo analyzing task...           │  │
│  └──────────────────────────────────────────┘  │
└─────────────────────────────────────────────────┘
```

#### 3.2.2 각 구역의 목적

**WORK ZONE (작업 구역)**
- 목적: 각 에이전트의 개별 책상
- 최대 수용: 3명 (기본)
- 기능: 에이전트가 독립적으로 작업
- 시각화: 도트 캐릭터가 정적으로 배치
- 초기 위치: 모든 에이전트의 기본 위치

**MEETING ROOM (회의실)**
- 목적: 에이전트 간 협업/토론 공간
- 최대 수용: 2-3명
- 기능: 동시에 여러 에이전트가 상호작용
- 시각화: 캐릭터들이 가까이 배치, 밝기 변동 애니메이션
- 활용: 크고 복잡한 작업의 협업 단계

**LOUNGE (라운지)**
- 목적: 휴식/대기 공간
- 최대 수용: 제한 없음
- 기능: 캐릭터가 쉬고 있는 상태 표현
- 시각화: 느슨한 배치, 천천한 애니메이션
- 활용: 에이전트 스트레스 감소 (v2.0+)

#### 3.2.3 구역 인터페이스

```
┌─ WORK ZONE (빨강) ─┐
│ ◼◼ WORK ◼◼         │
│ ┌─────────────┐   │
│ │  [■]  [■]  │   │
│ │             │   │
│ └─────────────┘   │
│ [▶ TO WORK]      │
└────────────────────┘
```

각 구역:
- **제목**: 구역 이름 + 박스 문자
- **격자 영역**: 에이전트 캐릭터가 표시되는 공간
- **액션 버튼**: 해당 구역으로 이동하는 버튼

### 3.4 실시간 작업 및 로그 시스템

#### 3.4.1 Global Mission Input
사용자가 최상위 임무를 입력하는 인터페이스:

```
┌─ [MISSION INPUT] ────────────────────┐
│ [입력창: Enter mission...          ] │
│ [◄ EXECUTE ►]                        │
└──────────────────────────────────────┘
```

기능:
- 텍스트 입력 필드
- Enter 키 또는 버튼으로 실행
- 실행 중: 버튼 비활성화
- 미션 길이 제한: 최대 100자 (표시는 40자)

#### 3.4.2 Agent Chat Interface
에이전트들의 상태와 활동을 표현:

```
┌─ [TERMINAL LOG] ─────────────────────┐
│ > SYSTEM INITIALIZED                 │
│ > AGENTS LOADED [3/3]                │
│ > [MISSION] React 컴포넌트 라이... │
│ > LEO: ANALYZING                     │
│ > MOMO: COLLABORATING                │
│ > [COLLAB] DISCUSSION ACTIVE         │
│ > [COMPLETE] MISSION SUCCESS         │
└──────────────────────────────────────┘
```

로그 포맷:
```
> [SYSTEM] 시스템 메시지
> [MISSION] 사용자 입력한 미션
> [COLLAB] 에이전트 협업 메시지
> [COMPLETE] 완료 메시지
> AGENT_NAME: 액션 설명
```

기능:
- 최대 5-6개 로그 라인 표시 (스크롤 가능)
- 자동 스크롤 (최신 로그 보임)
- 터미널 스타일 폰트 (Courier New)

#### 3.4.3 Agent Status Panel
각 에이전트의 현재 상태 표시:

```
┌─ [AGENTS STATUS] ────────┐
│ LEO (빨강)               │

│                          │
```

## 5. 사용자 경험 (UX) 시나리오

### 5.1 기본 사용 플로우

```
1️⃣ 시스템 시작
   └─ 대시보드 열기
      • 3명의 에이전트가 WORK ZONE에 배치
      • 모두 IDLE 상태
      • 터미널 로그에 시스템 초기화 메시지

2️⃣ 미션 부여
   └─ 입력창에 "Next.js로 투두 리스트 앱 만들기" 입력
      • [EXECUTE] 버튼 클릭 또는 Enter 키

3️⃣ 에이전트 협업 프로세스 (자동 진행)
   
   시간 0초: 미션 입력 처리
   └─ [로그] > [MISSION] Next.js로 투두 리스트...
      [로그] > LEO: ANALYZING
   
   시간 0.3초: LEO가 회의실로 이동
   └─ LEO의 status: IDLE → MOVING
      애니메이션: jumping (0.6초)
      위치: WORK ZONE → MEETING ROOM
   
   시간 0.7초: MOMO가 회의실로 이동
   └─ MOMO의 status: IDLE → MOVING
      [로그] > MOMO: COLLABORATING
   
   시간 1.1초: 협업 시작
   └─ LEO, MOMO의 status: COMMUNICATING
      애니메이션: talking (밝기 변함)
      [로그] > [COLLAB] DISCUSSION ACTIVE
   
   시간 2.8초: 미션 완료
   └─ [로그] > [COMPLETE] MISSION SUCCESS
      모든 에이전트: IDLE 상태로 복귀
      MEETING ROOM → WORK ZONE 이동

4️⃣ 결과 확인
   └─ 상태 패널에서 모든 에이전트 IDLE 확인
      터미널 로그 검토
```

### 5.2 인터랙션 세부사항

#### 5.2.1 에이전트 이동 버튼
각 구역 하단에 이동 버튼:
```
[▶ TO WORK]     [▶ TO MEETING]     [▶ TO LOUNGE]
```

동작:
- 클릭 시 해당 에이전트가 그 구역으로 이동
- 이동 애니메이션: jumping (0.6초)
- 위치: 구역 내 랜덤 좌표 (충돌 방지)

#### 5.2.2 미션 실행 흐름

```
[사용자 입력]
    ↓
[EXECUTE 버튼]
    ↓
[상태 변경: executing = true]
    ↓
[타이밍 기반 애니메이션]
    0ms:   미션 로그 출력
    300ms: LEO 움직임
    700ms: MOMO 움직임
    1100ms: 협업 시작
    2800ms: 완료 + executing = false
    ↓
[다음 미션 입력 가능]
```

### 5.3 사용자 기대효과

```
Before (텍스트만)          After (픽셀 아트)
─────────────────          ──────────────────
> Task started             [■LEO 움직임]
> Agent 1 working          [■MOMO 움직임]
> Agent 2 working          [협업 애니메이션]
> Completed                [로그 + 시각화]

❌ 진행도 파악 어려움       ✅ 진행도 직관적
❌ 감정적 연결 부족         ✅ 캐릭터와 공감
❌ 모니터링 지루함          ✅ 재미있게 관찰
```

## 6. 데이터 모델 (Data Model)

### 6.1 Agent 스키마

```typescript
interface Agent {
  id: 'leo' | 'momo' | 'alex';
  name: string;                    // 'LEO', 'MOMO', 'ALEX'
  role: string;                    // '코드 마스터' 등
  zone: 'work' | 'meeting' | 'lounge';
  x: number;                       // 구역 내 X 좌표 (0-300px)
  y: number;                       // 구역 내 Y 좌표 (0-150px)
  status: AgentStatus;
  color: string;                   // 16진수 색상 (#ff4444)
  stressLevel?: number;            // v2.0+: 0-100
}
```

### 6.2 AppState 스키마

```typescript
interface AppState {
  agents: Agent[];                 // 3명의 에이전트
  logs: string[];                  // 터미널 로그 (최대 6개)
  missionInput: string;            // 현재 입력한 미션
  executing: boolean;              // 실행 중 여부
  startTime?: number;              // 미션 시작 시간 (ms)
}
```

### 9.1 기술적 위험

| 위험 | 영향 | 완화 전략 |

| 위험 | 영향 | 완화 전략 |

### 9.3 비즈니스 위험

| 위험 | 영향 | 완화 전략 |
|------|------|----------|
